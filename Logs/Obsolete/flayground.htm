ある程度は自分の都合を考えつつ、エンドユーザーがさわるときに不便をさせないというバランスを考えるにおいて、次の変更を施す。

* srcZip によるアーカイブの作成時に README.md が除外される設定をやめる
* プロジェクトのバイナリーやソースの利用者が知るべきことの全てを README.md に入れる

数日前にいったん全ての README.md を消した。人に使っていただく考えのないごく小規模の個人プロジェクトに大げさにそういうのをつけても誰得で、管理コストが発生するデメリットしか見えなかったため。

しかし、GitHub でプロジェクトを公開するにおいては作法なのだろうから一応は適当に。

流れとしては、次のようになりそう。

* README.md もその都度見て更新
* ソースのアーカイブを作り、Resources/Archives に入れてコミット
* flayground など、「ソースとバイナリーを含むアーカイブ」とは別に「バイナリーが生成するファイルのアーカイブ」などがある場合には後者を Releases に追加
* その際、これらの一連のアーカイブを作る間に別のコミットを行わず、GitHub が自動的に生成する Source code (zip) などのアーカイブの内容が、自分が作ったソースのアーカイブのものと一致するようにする

* ファイル名により内容が分かる
* 具体的な表現なので、-photos など、ほかにも作れる

そのときには、それらが全体として何なのか、どう機能するのか、といったところに基づいて表現を選べばよい。

さらに、.gitignore もアーカイブに入るように。GitHub が自動生成するアーカイブに入るため。

flayground は、ただの生成ツールであり、フォントそのものでないため、GPL v3 でよい。

フォントの含まれる方のアーカイブには、元のフォント2種類のライセンスファイルを入れておく。

ttfautohint にもライセンスがあるが、これを同梱・再配布するわけでないため、フォントに自動的に入るメタデータを（作法として）残しておけばよいだろう。

-fonts のファイル名に変更する分で、

* 自分がアーカイブしたソース
* リリースの追加時のコミットのソース丸ごととして GitHub が自動的にアーカイブするもの

の二つが不一致になりそう。実害がないため直さない。

全てのプロジェクトの README.md を一気書きするときに全て再アーカイブするため、そのときに一致する。

✓ 現行の全てのプロジェクトのアーカイブの名前を更新し、Resources/Archives に移動 ∞
✓ GitHub にリリースを追加 ∞

本文の書き方にコツが必要。文章を最小限とし、できるだけリストで簡潔に。

使用させていただいたフォントやツール:
UDEVGothic_v1.2.1.zip
PlemolJP_v1.5.0.zip
ttfautohint-1.8.4-win32.7z

だけでよい。

タグを v0.1 と最小限にすれば、自動生成されるソースのアーカイブの URL が

https://github.com/nao7sep/flayground/archive/refs/tags/v0.1.zip

になる。

ファイル名にプロジェクト名を入れてほしいが、タグの方に入れるとプロジェクト名と重複になる。

✓ 新しい構成でソースをダウンロードしたりリリースを利用したりし、エンドユーザーから見て「ふつう」になっているか調べる ∞

GitHub 側においてプロジェクトのところに Archived がなく、Resources/Archives に集約されているのは、以前よりスッキリ。

ソースのアーカイブの名前は flayground-v0.1-20230427T033652Z.zip。

リリースのところには、

* flayground-v0.1-fonts.zip
* flayground-0.1.zip
* flayground-0.1.tar.gz

がある。

そのうち最後の二つに v が入らないのは、v*.*.* を推奨していながらも v が入っていたら削る GitHub の仕様か。

GitHub は主にソースを扱うところであり、「プロジェクト名＋最もシンプルな形でのバージョン番号」となっていればそれがソースであることは広く認知されているだろうから、無理やり v を入れる試みはしない。

フォントのアーカイブにはライセンスファイルが二つ入っている。

それぞれの名前と内容をチェックした。

GitHub が自動生成する二つのソースのアーカイブの内容が一致。

自分が生成したソースのアーカイブと GitHub が自動生成したものの違いがいくつかある。

そもそも前者は古い。いずれ README.md を追加するときに新しいアーカイブを作るし、古くてもフォントファイルの内容には影響がないため、そのまま。

違いとしては、前者のみ Compiled を含み、後者のみ .gitignore と LICENSE を含み、srcZip.txt の内容に新旧の違いがあり、テキスト系ファイルにおいては改行が Windows 形式か Unix 形式かも異なっている。

README.md の追加時に改行コード以外は一致する。

Plemol の方で "39" にしたつもりの部分が "35" になっていたので修正する。

今さら名前がアホっぽいと思い始めている（笑）。「フォントでいろいろと遊ぶところ」なので、font と playground から2秒で flayground とした。変更がめんどくさいので見送るが、次からはもう少しまじめに名前を考えないといけない。

✓ 元のフォントが更新されていないかチェック ∞

どちらも最新版。

✓ Plemol の名前のミスを修正 ∞
✓ Plemol* を Plemo* に変更 ∞

PlemolJP から Plemol とするのはパクりすぎな気もしていた。

PreMo とか、ちょっとがんばることも考えたが、あとで「やっぱり」と思いそうなので L だけ削る。

✓ 新旧の .ttx ファイルを比較 ∞
✓ アーカイブ ∞
✓ リリースを追加 ∞
