bom

思いも寄らないところに Shift-JIS のファイルが入る。また、UTF-8 になっていても、一部の（外国の）アプリは BOM がないと Windows-1252 だと誤検出するなどがある。

しょーもないことなので無視してきたが、今度は README.md が Shift-JIS でつくられていて、VSC はスマートに判別するのでステージし、GitKraken で最終チェックしようとしたときに文字化けしていて気づいた。

自分は困らないが、GitHub などにファイルを載せて公開していくにおいては誰かが困りうる。この機会にパッとツールをつくる。

✓ プロジェクト開始時のタスクを処理 ∞
✓ 落ちるところがないようにする ∞
✓ タスクリストのログが出力されるようにする ∞
✓ README.md を更新 ∞
✓ 実装 ∞
✓ テスト ∞
✓ コーディングのガイドラインに準拠しているか調べる ∞

大丈夫そう。

✓ プロダクトとしての完全性を高める ∞

実装が終わり、テストが終わり、だからこそ README.md にも初期的な内容を書き込んでコミットしたのだろうが、それから少し時間が経ったので、コードを全体的にチェックし、いったん使用フェーズに移行できる程度にはまとまっているのを確認した。

UTF-8 なのに BOM が入っていなくて GitKraken で化けたファイルがいくつかあったのが開発の理由。BOM が入っていなくても UTF-8 にフォールバックするのが昨今のスタンダードな実装だが、非 CJK 圏のツールには、そうでないものがちょくちょくある。

✓ アーカイブ ∞

Python のスクリプトでビルドおよびアーカイブを行った。このスクリプトによりアジャイルな開発に移行できるので、今後はアーカイブについてはタスクに入れなくてよいと思う。
