yyChat を急ぐべきだが、娘の受験や今すぐに急ぐ仕事などがあって、ちょっと追いつめられている。ここ数週間、睡眠時間がずっと5時間くらいだ。平日2時から5時まで図書館で娘に勉強を教えているので仕事の時間が激減している。いずれ yyChat に流し込めるデータをいったんレポジトリーに保存しておき、奥さんの側でレポジトリーのファイルを更新すれば HTML ファイルで閲覧くらいはできる簡易的なツールをつくる。それでも一日の終わりに同期作業をすることで情報共有は劇的に良くなる。

いずれは yyChat に会社の業務を統合する。しかし、そちらには、ユーザー管理、セキュリティー、GET/POST に頼り過ぎない UI などが必要で、朝から晩までフルで開発だけやれる日が続いたとしても10営業日は絶対にかかる。一方、今の自分は、娘の受験勉強、会社の業務、9月のベトナムへの準備などが重なっていて、ろくに眠ることもできていない。どう考えても、yyChat はすぐにはつくることができない。

そのため、yyChat に使えるくらいきちんとしたローカルモジュールをつくり、そこに決め打ちのユーザー ID で、単一ユーザーとしてメモ、ファイル、写真、URL などをタスク単位かつ時系列的に放り込んでいけるコンソールアプリをつくる。

タスクについてできるべきは、追加、属性の修正、削除、アーカイブだ。

それぞれに対し、クリップボードのテキストの読み込み、文字列の直接入力（しかし複数行にはできない）、ファイルや画像の添付を可能とする。

おそらく、全てがテキストエントリーであるとみなし、URL もファイルも写真もそれらにさらに添付できるのが望ましい。そうでないと、コメントを書いてから写真の人と、写真を載せてからコメントの人の違いにより、共有データが分かりにくくなる。

URL は、本文から抽出されるので十分か。個別に書き添えるのだと本文のどの部分との関連があるか分かりにくい。

流れていくのが大事なので、たとえば1～10を載せたかったのに5を載せ忘れたから4と6の間を選択してそこに挿入……のようなことを実装しない。実装がめんどくさいし、使うのもめんどくさいし、見る側も「知らないうちに、そんなところに入っていた」というのが分かりにくい。10の直後、あるいはしばらく経ってからの「ごめん、入れ忘れた」で済む。

yyChat にも関わってくることだが、左側メニューの分割においては、取引先単位より、「終わりのある活動」を単位とするのが適するだろう。それが中長期のプロジェクトなら、無理にでも細かく割ればいい。早ければ数日、遅くても数ヶ月で終わるくらいの単位に分割し、ちゃんと進めてちゃんと終えてアーカイブする。まさにソフトウェア開発のチケットシステムで、慣れているのでうまく回せる。

そのログの表示方法としては、この一時的なツールにおいては、プロジェクトごとの表示はなくてよい。あったほうが便利なときがあるのは間違いないが、時間が限られているなら、「統合ビュー」的なものだけで夫婦間は足りる。自分の側でレポジトリーにファイルを入れていき、奥さんにレポジトリーのファイルを更新してもらう。HTML ファイルは一つだけで、それが大きすぎるようになるまでに yyChat を完成させてデータを移行する。
