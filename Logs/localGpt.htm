localGpt

ChatGPT とウェブやアプリでやりとりすることの不便の一つは、データの再利用に向かないこと。

アイデア出しを手伝ってもらってすぐに会話を終えるようなときにはウェブやアプリで十二分に便利だが、開発などにおいてなぜそう決めたのか根拠を残したいとき、いったん会話を終えるものの必要に応じて確実に再開したいとき、特定の会話そのものをブログなどの記事にしたいときなど、やりとりをより継続的に活用したいときには、すぐログが散らかるウェブやアプリでは困る。

ChatGPT は、ある時期での情報のカットオフを勘案しても、最強の調べ物ツールだ。

さらにウェブの情報を検索するようには API ベースでも実装できるようだが、どういう検索をかけ、得られた HTML からいかに不要な部分を削り、それらをそれまでのいくつかのメッセージとどのように統合してコンテキスト化し、どう整理するかといったところにはノウハウが多々あるはずで、そこまでつくり込んでは開発コストが一気に大きくなる。

5～10ほどのページを読んだとして、特定のものが極端に大きいとか、ほかとあまりに異なることを言うページがあるとか、ページの内容が派閥（？）に分かれていて A 派と B 派が認められるとかにおいて、さらにいくつかのページは個人ブログで、ほかのいくつかのページはフォーラムだから不特定多数の人の短いメッセージで……といった無数のイレギュラーに自分一人のコードで無難に対処することは極めて難しい。

OpenAI のカットオフに任せた方が、それまでの期間の学習データの質が一定以上に保たれるであろう点においても合理的だ。

プロンプトを @ から始めるなどの簡単な工夫により画像も生成させたい。けっこうハズれるので、外れたら Backspace キーか何かで、その生成自体をなかったことにする。

プロファイルの指定により system メッセージを設定できるなら、「これはブログ用のやりとりだから、こういうふうにしてほしい」だったり、「これは調べ物だから、質問あるいは回答に重複が認められたら、回答をやめ、すぐさま警告してほしい」だったりをプロファイルで指定してのやりとりを洗練していくことができる。

また、やりとりの流れそのものを工夫することも可能で、たとえば、こちらからのプロンプトをまとめ直して含める形で回答を生成し、しかもその全体にタイトルを設定して「記事」の集合を出力してもらうなども可能。このモードでつくった JSON ファイルなら、「記事」のタイトルを左側メニューに置いた HTML ページとすることで、ちょっとした参考資料になる。

ChatGPT のアカウントが人数分なくても複数人に API ベースで ChatGPT を使ってもらえるのも強み。

✓ ナレッジベース的な運用を考えすぎない仕様に変更 ∞

AI によるタイトルの自動生成というのが引っかかっている。コントロール不能で、歩留まりになりそう。絶対に MECE なエントリーにはできない。それなら、本家と同様、内容の重複を過度に問題視しない、人間の認知において気軽に使えるツールにしてしまった方が使用時に疲れない。

JSON でやるというのは、データを貯めてツールによりナレッジベースにするなどを望んでのことでもあるが、それにより複数のカテゴリーに同じような内容のエントリーが入っては、それもまた「重複」「冗長」だと認識して積極的に改めるべきだろうか。

ナレッジベースそのものがプロダクトなら、その管理に時間を割くのは仕方ないコストだが、今のゴールはあくまでコーディングだったりビジネスだったりだ。
