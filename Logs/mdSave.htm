splitQuickNotes を数日前に書いた。それと整合するフォーマットで Markdown ファイルを出力するツールを書く。それから、今現在 QuickNotes で更新しているファイルを消す。

ファイルのロードによる編集機能の実装も考えたが、VSC だと差分をチェック・ステージしながらできることなので不要。日によっては目が疲れているなどで、より大きなフォントや別のフォントで編集したいときもある。それには Mery を使うだろう。差分を見られる VSC と、たまに気まぐれでフォントにこだわれる Mery にない付加価値をつけにくい。

Avalonia UI で書いたら Mac でも使える。そこも少し考えたが、しょせん平文なのでテキストのみ同期すれば足りる。たとえば図書館に娘を連れて行くなどで（モニターなどにつながっている Windows のノートを取り外したくなくて）Mac を持参し、気が向いて何か書けば、わざわざテザリングにつないですぐに同期するのはめんどくさいので、すぐ分かるファイル名をつけた記事を OneDrive の TODO ディレクトリーにでも保存し、急ぐなら携帯から自分に TODO メールを送ればよい。帰宅後、Mac を開けば Wi-Fi により OneDrive が同期され、Windows パソコンでシームレスに開ける。

そのうち TODO メールのシステムを書くと決めている。自宅で Mac なら、TODO メールとして自分に送ってしまうのが、それをあとでチェックして掲載するまでを忘れない点において効率的。

いずれにしても、Mac のみでワークフローを完結させる必要性が乏しいし、そもそも Mac はネトフリ専用機になっていて文字を入力する機会がとても稀だろうから、WPF でサクッと書く。

QuickNotes と同じことをやる。GUID のみ、入れるかどうかをチェックボックスで指定できるようにして、デフォルトでは入れない。ブログ記事には必ずタイトルをつけるが、QuickNotes との仕様の整合性のため、タイトルを optional にする。なければファイル名のタイムスタンプのあとに空白が入らないのも同じ。

splitQuickNotes と違って構文解析が不要なので、バイナリーの配置まで2時間もかからないはず。もうすぐ年越しなので開発は来年になるが、覚えていれば所要時間を計測してみる。単機能で特にアルゴリズムも不要なツールの開発にどのくらいの時間がかかるのか知っておきたい。

自動保存もやめておく。短い記事ならツールでそのまま書いてしまうかもしれないと思ったが、編集を実装しない手抜きをするなら、結局、VSC や Mery で「無題」的なファイルをつくって、たまに保存しながらザーッと書いて、最後にツールに貼りつけて保存のみ行うワークフローが定着する。
