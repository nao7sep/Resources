✗ taskKiller との親和性について考える ∞

「～について調べる」のタスクのメモにいきなり回答を入れると、多言語ではややこしくなる。タスクを Done などにしたあとに追加したくなることも多かった。taskKiller のほうは、あくまで自分の言葉で自分の考えを。AI の回答でログを肥大化させない。

回答に対する GUID が絶対 URL の一部なら taskKiller のログからリンクを張れる。ベース URL を変更できるなら、URL の変化にも対応できる。ただ、それも本質的には二重管理。

taskKiller のログとこのアプリのデータはリンクしないものとする。コードと関連付けるし、できるだけテーマごとにファイルを作るため、「調べた」という事実が分かっていれば、そのデータがどこにあるかは低コストで分かるし、そうなる使い方に慣れていくだけのこと。

✗ コマンド体系を考える ∞

せっかく AI が英語を直してくれるので、コンテキスト情報も校正してもらって保存したい。

思い付きだが、

con

で「コンテキスト追加」モードになり、単一行で書いたことを修正してくれ、それを追加するかどうか y/n で聞かれる。入力せず Enter または y/n で n によりモードが終わる。何度も聞かれても使いにくい。

AI に直してもらうので、コンテキストの修正はやりにくい。

表示と削除については、追加より頻度が低いので「con に続けて何か」でよい。con list / con del でよいか。

スマートに実装する。con list で仮の連番を付けて表示。del 直前に list をやっていれば番号のみ聞いてくる。やっていなければ list をやってくれる。con del * で番号を聞かれずいきなりの指定も可能に。連番の選択ミスがありうるので、一応確認。

それなら con * でいきなり追加もやる。con を con add の糖衣構文扱いする考えもあるが、このシンプルなコマンド体系においては冗長。

✗ コンテキストが概ね一致するならファイル間で回答を移動できるようにする ∞

ディレクトリー分けするため、ファイル名のみ与えれば同一ディレクトリー内で、パスを与えればそれを特定。

「何を、そこへ移動し、先頭なのか末尾なのか特定のエントリーの前後なのか」を指定。

この処理は、ファイルを構文解析し、エントリー単位に分割し、エントリーを操作し、ファイルを上書きした場合に変更点以外が必ず一致するラウンドトリップ性の高いフォーマットを必要とする。

管理が複雑なのと、タグ付けの方が便利そうなので、複数ファイルの扱いについて迷っている。単一ファイルに全て詰め込み、リリース時にツールで HTML ページに変換してアーカイブに含めるのも一つの方法。同じ差分が二度、目に入る二重管理を避けたい。Compiled ディレクトリー的なところに入れるなら問題にならない。

ワークフローがシンプルでないと管理コストがかさむ。質問に答えるのは AI にとっては一瞬のことなので、複数のプロジェクトで同じような情報が必要になったなら、それぞれで同じような質問を、できるだけコンテキストを意識しながら聞き直すので十分。それぞれのファイルがそれ単独でそのプロジェクトに必要な情報を含むようにすればよい。
