yyGptLib

質問をタイトルっぽく整理してもらうのは Custom instructions でできている。それから答えてもらい、タイトルも回答も英語と日本語の両方でローカル保存するまでは API で間違いなく可能。

たとえば新しいクラスライブラリーを作るにおいて、「こういう機能を入れたいのだが、どういう名前空間が必要か」を会話でやり取りするなら、そのやり取りを /Source/General.gpt にでも入れ、文字列処理に関することを /Source/StringManipulation/General.gpt に入れ……といったことは、ある程度のトレーサビリティーが生じる点において合理的。

ただ、そのうち StringManipulation 名前空間を二つに割るなどがあれば、そこの General.gpt は容易には割れず詰む。

かといって、そういうカテゴリー分けを最初から諦めて時系列的に全て詰め込むのも問題がある。

そのあたりのことから考える必要がある。

ローカル版とリモート版の両方に長短があるので、両方を作る？

ローカル版だとレポジトリーに入れられる。開発関連の情報に向く。

リモート版だと、API 連携により情報の整理・共有のプラットフォームに流し込める。

✓ GPT-4 Turbo を試す ∞

長文を一発で翻訳できる。ただの GPT-4 だと意味が大幅に削られる。まだプレビュー版なので、決め打ちにはできない。モデル名も設定ファイルなどで指定できるようにする。

全体的に簡略化する。余計なことを考えすぎていた。

たとえば、回答の順序を並び替える必要はない。すでに答えてくれていたことを自分がよく分かっていなくて再度聞くとか、最初に聞いたときには教えてくれなかったことがあとから出てきて、あとで読むならそちらを先に読んだ方が良いとかのグダグダはいくらでも起こりうる。そもそも回答は MECE にならない。そのため、「一応はタイトルをつけて記事単位で表示できるが、あくまで会話の一部として存在する」というのがコンセプトのツールだとの位置づけが合理的だ。

GUI での作業を中心とするなら、diff を取れないのは問題でない。そもそも回答を編集できなくする。質問し、回答を得て、その回答が気に入れば残し、そうでないなら消して聞き直す。残すものなら、それが改変されていないということがハッシュ値により保証されるので、その diff をチェックすることなく、そのままコミットすればよい。見るとしても、消そうと思ったものを消し忘れていないか程度で十分だ。

自分は何においてもコミット前の diff のチェックが習慣になっていて、このプログラムの設計でも最初は diff に向くファイルフォーマットを考えた。しかし、AI の回答については、生成されたものが改変されていないということのみ確認できていれば、その一字一句を自分が見る必要性は当然ない。読むのもザッとで、タスクやアイデアの取りこぼしもある。何度も聞き、量から質を高めていくアプローチになる。diff を取れない JSON にするのは、そういった割り切りも考えてのこと。

✓ コーディングのガイドラインに準拠しているか調べる ∞
✓ エンドポイントやモデル名をライブラリー側で設定できるようにする ∞

yyLib で chat_endpoint と chat_model を指定できるようにした。これらを設定し、上位のアプリでこれらを設定しなければ、共通設定が全てのアプリで使われるようにしていく。

yyGptLib では、yyGptChatConnectionInfoModel で endpoint を元から指定できる。これは GptChatConnection.json のようなファイルからロードされる。ここに model も入れてしまいたいが、これは「接続」に関する情報の一部でない。

yyGptLibConsole は、テスト用に model を gpt-4 にする。もう yyGptLib のデフォルトを gpt-3.5-turbo から変更してもよいかもしれないが、今後もモデルは更新されていくので、コード内のリテラルをあまりさわりたくない（追い続けることになる）。共通設定や上位アプリの設定で何とかすること。
